# 任务管理工具需求规格说明

## 1. 文档目的

本文档用于在新一轮方案重置后，重新定义任务管理工具的产品目标、范围、术语、功能需求、业务规则与非功能约束，作为后续详细设计与实施计划的唯一需求基线。

本文档只回答“要做什么”和“要满足什么”，不预设具体技术栈、模块划分或实现细节。

## 2. 产品概述

本产品是一个面向个人长期使用的轻量级任务管理工具，核心关注点不是日程占用，而是任务是否能在截止时间前完成。

产品遵循以下基本理念：

- 任务管理以截止时间为核心。
- 开始时间表示任务从何时开始进入用户视野，不表示该任务在该时间段内持续占满。
- 用户不需要传统意义上的“范围任务”或“持续事件”建模。
- 对跨天任务，用户最终只关心是否在截止时间前完成，不强调具体是哪一天完成。
- 重复任务需要支持未来延续，同时允许对某一次单独调整。
- 同步语义优先保证一致性和简单性，而不是复杂的双向冲突合并。

## 3. 产品目标

- 提供一个适合个人长期使用的截止时间导向任务工具。
- 支持单次任务与重复任务。
- 支持危险日提示与截止风险提醒。
- 支持日历视图与近期视图两种核心查看方式。
- 支持本地优先使用，并通过远端同步实现多端数据一致。
- 支持 Windows 与 Android 双端兼容使用。
- 尽量降低后台驻留时的资源占用。

## 4. 范围与非目标

### 4.1 本期范围

- 任务创建、编辑、删除、状态修改
- 日历视图
- 近期视图
- 重复规则
- 模板修改与单次覆盖
- 危险日规则
- 工作日与节假日处理
- 标签与颜色
- 备注
- 远端同步
- Windows 后台驻留与提醒

### 4.2 明确不做或暂不支持

- 多人协作
- 富文本备注
- 多标签
- 一个任务配置多个危险规则
- 传统“持续事件/整段占用型日程”建模
- 复杂双向冲突合并
- 浏览器在线版作为本期目标形态

说明：

- 若用户需要跳过某次重复任务，本质上等价于将该次实例标记为“取消”。
- Windows 与 Android 都属于当前目标平台，允许分阶段实现，但需求、设计与数据语义必须从一开始保持双端兼容。

## 5. 平台与阶段目标

- 目标平台为 Windows 与 Android。
- 两端都属于正式支持范围。
- 允许按阶段安排开发优先级，但不得在需求和设计层面把 Android 视为可选兼容项。
- Windows 端需要支持常驻后台、提醒和低资源占用。
- Android 端需要保证核心任务能力、同步语义与数据解释一致。

## 6. 核心术语

### 6.1 任务

用户需要完成的一项事项。任务以截止时间为核心，可带有开始时间、标签、优先级、备注、状态和提醒规则。

### 6.2 单点任务

只有一个明确时间点的任务。实现上可视为开始时间与截止时间相同的任务。

### 6.3 截止型任务

具有开始时间和截止时间的任务。开始时间表示任务进入视野的时间，截止时间表示最晚完成时刻。

### 6.4 任务模板

用于定义重复任务规则的基础配置。模板负责生成未来任务实例。

### 6.5 任务实例

单次任务本身，或由重复模板展开后形成的某一次具体任务。

### 6.6 单次覆盖

对某个具体任务实例进行的独立修改，不影响同模板下其他实例。

### 6.7 危险日

根据截止时间按规则倒推得到的单个提示时间点，用于提示任务已接近风险区间。

### 6.8 工作日计算

在危险日倒推时，可选择按工作日而非自然日计算，并结合节假日与调休数据跳过非工作日。

### 6.9 状态

任务状态固定为以下三种：

- 未完成
- 完成
- 取消

## 7. 用户与使用场景

### 7.1 目标用户

- 需要长期规划个人事项的用户
- 有较多周期性任务的用户
- 需要围绕截止时间进行预警的用户
- 主要在 Windows 上使用，并希望后续可扩展到移动端的用户

### 7.2 核心使用场景

- 记录一个未来某时刻要处理的单次事项
- 记录一个有开始时间和截止时间的任务，并在截止前完成
- 创建按小时、天、周、月、年重复的事项
- 为任务配置危险日，在临近截止前突出显示
- 针对某次重复任务单独延期任务时间或危险日
- 浏览全部任务的时间分布
- 快速查看未来 31 天内最需要关注的任务
- 在多端之间保持数据一致

## 8. 功能需求

### 8.1 任务基础信息

系统必须支持以下任务字段：

- 标题
- 备注
- 开始时间
- 截止时间
- 是否全天
- 标签
- 优先级
- 状态
- 重复规则
- 危险日规则

具体要求如下：

- 标题为必填。
- 备注为纯文本，支持长文本与换行。
- 任务可配置开始时间和截止时间。
- 单点任务允许开始时间与截止时间相同。
- 全天任务按日维度处理。
- 非全天任务最小精度支持到分钟，详细精度在设计阶段明确。
- 标签为单选。
- 每个标签对应一种颜色。
- 优先级固定为三档：低、中、高。
- 任务默认状态为“未完成”。
- 任务只能被手动标记为“完成”或“取消”。

### 8.2 日历视图

系统必须提供日历视图，作为全量任务的主展示界面之一。

要求如下：

- 以纵向时间流方式展示所有日期。
- 日期范围支持长时间跨度浏览。
- 视图支持持续滚动查看。
- 同一天中的所有任务应尽量完整展示，不做默认折叠。
- 单点任务以单日任务样式展示。
- 带开始时间和截止时间的任务，在查看层面需要体现其进入视野时间和截止时间。
- 任务颜色按标签颜色展示。
- 存在危险日的任务，在危险日对应日期应采用更醒目的样式。
- 已完成和已取消任务仍需显示，但样式弱化。

### 8.3 近期视图

系统必须提供近期视图，用于展示从当前日期开始、未来最多 31 天内的任务。

要求如下：

- 近期视图与日历视图并列为两种主要展示方式。
- 展示范围默认为当前时刻至未来 31 天。
- 展示所有状态的任务。
- 未完成任务应优先展示。
- 已完成和已取消任务继续显示，但排在后部并采用弱化样式。
- 排序应同时考虑静态优先级与危险日临近程度。

### 8.4 重复任务

系统必须支持重复任务。

要求如下：

- 支持按小时、天、周、月、年重复。
- 支持以上单位的 n 倍重复。
- 支持无限循环。
- 支持配置循环截止日期。
- 支持未来任务的模板修改。
- 支持仅修改单次实例。

### 8.5 模板修改与单次覆盖

系统必须支持重复任务模板与单次实例并存的修改机制。

要求如下：

- 用户可修改任务模板并影响未来实例。
- 用户可仅修改某次实例，且仅影响该次实例。
- 模板修改不影响历史实例。
- 模板修改应支持设置生效日期。
- 模板修改时，生效日期默认建议为下次任务开始时间。
- 已被单独修改过的未来实例，在模板修改后默认保留单次覆盖。
- 用户可选清除未来实例上的既有覆盖。

### 8.6 状态管理

系统必须支持三态状态管理。

要求如下：

- 默认状态为未完成。
- 用户可手动标记任务为完成。
- 用户可手动标记任务为取消。
- 对于重复任务，“跳过本次”等价于将该次实例状态改为取消。
- 已完成任务在展示上应颜色变淡。
- 已取消任务在展示上也应颜色变淡，但不得直接隐藏。

### 8.7 危险日规则

系统必须支持单一危险日规则。

要求如下：

- 每个任务当前只支持一个危险提醒规则。
- 危险日可按“提前几小时”或“提前几天”计算。
- 危险日为单个时间点，不是区间。
- 危险日默认围绕截止时间倒推计算。
- 危险日可选择是否考虑工作日。
- 若考虑工作日，则需要结合节假日与调休数据跳过非工作日。
- 用户可对某次实例的危险日进行手动延期或调整。

### 8.8 节假日与工作日

系统必须支持节假日和工作日数据参与危险日计算。

要求如下：

- 系统需要接入可逐年更新的节假日数据源。
- 节假日数据需支持法定节假日和调休工作日。
- 危险日按工作日倒推时，应以节假日数据为准。

### 8.9 标签与颜色

系统必须支持标签与颜色管理。

要求如下：

- 标签支持自定义名称。
- 标签支持选择颜色。
- 系统应提供一组默认易区分的颜色。
- 用户可自定义标签颜色。
- 日历和近期视图中的任务颜色应以标签颜色为基础。

### 8.10 同步

系统必须支持远端同步。

要求如下：

- 应用启动或进入主工作状态时，先执行同步。
- 默认采用远程优先策略，以减少冲突。
- 用户保存修改前，应先检查远端是否有更新。
- 若保存前检测到远端数据已变化，则本次修改作废，以远端数据为准。
- 同步规则应尽量简单明确，优先保证数据一致性。
- 本期不要求复杂双向冲突合并。

### 8.11 后台运行

系统必须满足以下后台运行要求：

- Windows 端支持常驻后台。
- 在后台运行时应尽量降低资源占用。
- 后台能力应不影响正常提醒和同步机制。
- 主窗口关闭后的后台行为应明确可控，不允许造成长期高占用却无明显收益。

## 9. 业务规则

### 9.1 时间语义规则

- 所有任务的核心判断依据是截止时间。
- 开始时间用于决定任务从何时开始进入用户视野。
- 带开始时间和截止时间的任务，不表示该段时间被该任务持续占满。
- 系统不以“持续事件”作为本期核心建模对象。

### 9.2 显示规则

- 任务不因完成或取消而消失。
- 状态变化通过颜色与样式区分。
- 同一天任务较多时，优先保证完整展示，而非折叠隐藏。

### 9.3 修改规则

- 历史实例不受未来模板修改影响。
- 未来实例既可能由模板生成，也可能存在单次覆盖。
- 单次覆盖优先级高于模板展开结果。

### 9.4 同步规则

- 同步语义以远端为准。
- 本地数据主要承担工作副本与离线缓存角色。
- 当本地修改与远端最新状态冲突时，应优先保留远端状态。

## 10. 非功能需求

### 10.1 性能

- 应用在 Windows 常驻后台时应尽量保持低资源占用。
- 日历视图应能够在大量任务场景下保持可用和可滚动。
- 同步与提醒不应明显阻塞主交互。

### 10.2 可维护性

- 需求术语和规则应在实现中保持一致。
- 重复规则、危险日计算和同步逻辑应在后续设计阶段单独模块化。
- 平台能力应与核心任务规则解耦，便于后续扩展 Android。

### 10.3 一致性

- 不同平台对任务状态、危险日、重复规则和同步结果的解释必须一致。
- 平台差异允许体现在交互形态上，但不能改变核心业务语义。

## 11. 验收要点

以下条件满足时，可认为需求实现基本符合预期：

- 用户可以创建单点任务和带开始时间、截止时间的任务。
- 用户可以为任务设置标签、优先级、备注和状态。
- 用户可以创建并修改重复任务。
- 用户可以仅修改某次重复实例。
- 用户可以为任务设置危险日，并支持工作日倒推。
- 日历视图可以连续浏览全量日期与任务。
- 近期视图可以展示未来 31 天内任务，并按规则排序。
- 完成与取消状态的任务不会被隐藏，只会弱化显示。
- Windows 端可以稳定后台驻留并触发提醒。
- 数据可通过远端同步在多端间保持一致。
